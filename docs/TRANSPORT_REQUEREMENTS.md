Техническое задание: Корпоративный сервис заказа транспорта (PWA)

1. Общее описание проекта
Цель проекта: Разработка системы управления корпоративным парком автомобилей (~400 машин) для обслуживания медицинских учреждений (клиник, лабораторий).
Платформа: /home/hronop/node/moment/
Основные участники (роли): 
- Клиент - уже реализовано в /home/hronop/node/moment/app/client
- Диспетчер - /home/hronop/node/moment/app/dispatcher
- Водитель - /home/hronop/node/moment/app/driver
- Пассажир (конечный получатель услуги) - /home/hronop/node/moment/app/passenger
- Администратор системы - /home/hronop/node/moment/app/admin

2. Технологический стек
 * Frontend & Backend: Next.js (React) — SSR/SSG, API Routes.
 * База данных: PostgreSQL.
 * ORM: Prisma или TypeORM (для работы с БД).
 * Real-time взаимодействие: WebSockets (Socket.io или Pusher) — критично для обновления координат и статусов заказов.
 * Карты и геокодирование: Google Maps API, Mapbox или Yandex Maps API (построение маршрутов, расчет времени, отображение авто).
 * Аутентификация: NextAuth.js (JWT токены).
 * Стилизация: Tailwind CSS.
3. Ролевая модель и функциональные требования
3.1. Роль: Клиент (Медицинское учреждение)
Интерфейс: Web (Desktop / Mobile).
 * Авторизация: Вход по логину/паролю (выдается Администратором).
 * Создание заказа:
   * Ввод адреса подачи (А) и адреса назначения (В) с подсказками (Autocomplete).
   * Выбор типа автомобиля (например: стандарт, минивэн, спецтранспорт для инвалидов-колясочников).
   * Указание времени: "на сейчас" или предварительный заказ.
   * Комментарий к заказу (например, ФИО пациента, номер палаты).
 * История заказов: Просмотр активных и завершенных поездок, их статусов и стоимости/отчетности.
3.2. Роль: Диспетчер
Интерфейс: Web (Desktop - панель управления).
 * Дашборд: Интерактивная карта с отображением всех активных машин (~400 шт.) и их статусов (Свободен, На заказе, Офлайн).
 * Управление заказами: * Получение новых заявок от Клиентов в реальном времени.
   * Автоматический или ручной поиск ближайшего подходящего автомобиля (с учетом типа авто).
   * Назначение заказа конкретному водителю.
 * Мониторинг: Отслеживание трека автомобиля во время выполнения заказа.
3.3. Роль: Водитель
Интерфейс: PWA (Mobile-first, установка на экран телефона).
 * Рабочая смена: Кнопки "Выйти на линию" / "Уйти с линии" (включает/выключает передачу GPS-координат).
 * Прием заказа:
   * Push-уведомление (через Service Worker) или визуальный алерт о новом заказе.
   * Таймер на принятие решения.
   * Кнопка "Принять".
   * Кнопка "Отказаться": При нажатии открывается модальное окно с выбором причины из справочника (например: "Поломка", "Не успеваю", "Заканчивается смена") + поле "Иная причина" (текстовый ввод).
 * Процесс поездки: Кнопки изменения статусов: "На месте" -> "Пассажир в машине/Начал поездку" -> "Завершил поездку".
 * Завершение поездки: При нажатии "Завершить", на экране смартфона водителя генерируется уникальный QR-код для оценки поездки.
3.4. Роль: Пассажир (без обязательной регистрации)
Интерфейс: Web (Mobile).
 * Пассажир наводит камеру своего смартфона на QR-код с экрана водителя.
 * Открывается web-страница (уникальная ссылка заказа).
 * Возможность поставить оценку (от 1 до 5 звезд) и написать текстовый отзыв.
 * Данные автоматически привязываются к ID водителя, ID заказа и Клиенту.
3.5. Роль: Администратор
Интерфейс: Web (Desktop).
 * Управление пользователями (создание/редактирование Клиентов, Диспетчеров, Водителей).
 * Управление автопарком (добавление машин, привязка водителей к машинам).
 * Управление справочниками (редактирование списка "Причины отказа").
 * Аналитика и отчеты (по клиникам, по водителям, рейтинг водителей на основе отзывов).
4. Требования к PWA (Progressive Web App)
 * Manifest: Наличие manifest.json для установки приложения на главный экран смартфона (Add to Home Screen).
 * Service Workers: * Кэширование статических ассетов для быстрой загрузки.
   * Обработка потери сети (Offline mode): если водитель въехал в зону без интернета, приложение не должно "падать", а должно кэшировать статусы и отправить их при восстановлении связи.

Данные о текущем положении (current_lat/lng в таблице Vehicle) постоянно перезаписываются, а нам нужно сохранять весь путь для разрешения спорных ситуаций, контроля отклонений от маршрута и аналитики.
Поскольку GPS-трекер отправляет точки очень часто, эта таблица будет расти очень быстро.
Вот структура необходимой таблицы и важные технические нюансы для ее реализации.
 Структура таблицы для хранения маршрутов (Треков)
    5.7 Track_Point (Точки маршрута поездки)
Эта таблица хранит «хлебные крошки» (каждую переданную координату) во время выполнения заказа.
 * id (BigInt, Primary Key) — Здесь лучше использовать автоинкрементный BigInt, а не UUID, так как записей будут миллионы, и BigInt индексируется и работает быстрее.
 * order_id (UUID, Foreign Key) — Привязка к конкретному заказу (поездке).
 * vehicle_id (UUID, Foreign Key) — Привязка к машине.
 * driver_id (UUID, Foreign Key) — Привязка к водителю.
 * latitude (Float / Double Precision) — Широта.
 * longitude (Float / Double Precision) — Долгота.
 * speed (Float, Nullable) — Скорость в момент фиксации точки (полезно для контроля превышений).
 * heading (Float, Nullable) — Направление движения (в градусах).
 * accuracy (Float, Nullable) — Погрешность GPS в метрах (позволяет отфильтровывать "прыжки" координат, когда телефон теряет спутники).
 * recorded_at (DateTime / Timestamp) — Время фиксации точки на устройстве (критично важно для офлайн-синхронизации, когда данные приходят на сервер с опозданием).
 * created_at (DateTime / Timestamp) — Время фактической записи в базу данных сервером.
Архитектурная ремарка (Candor check)
Для автопарка в 400 машин классический подход "просто писать в таблицу Postgres" быстро станет узким местом. Давайте посчитаем: если 200 машин одновременно находятся на линии 8 часов и отправляют координаты каждые 10 секунд, это генерирует около 576 000 строк в день (или ~17 миллионов строк в месяц).
Чтобы база данных не «легла» через полгода работы, программисту необходимо учесть следующие требования к БД:
 * Партицирование (Partitioning): Таблицу Track_Point в PostgreSQL необходимо изначально разбить на партиции (например, по месяцам ). Это позволит быстро удалять или архивировать старые треки (например, старше 3 месяцев), просто отключая партицию, а не выполняя тяжелый запрос DELETE.
 * PostGIS (Расширение для PostgreSQL): Вместо двух обычных колонок latitude и longitude, настоятельно рекомендуется использовать тип данных geometry(Point, 4326). Это позволит базе данных "понимать" географию и делать быстрые запросы вроде: "покажи все точки маршрута внутри вот этого полигона" или "какая реальная дистанция (пробег) получилась по этим точкам".
 * Индексы: Обязателен составной индекс по (order_id, recorded_at) для быстрого извлечения трека конкретной поездки для отрисовки на карте диспетчера.

6. Жизненный цикл заказа (Сценарий)
 * Клиент создает Order (Статус: Created).
 * Диспетчер видит Order, выбирает свободный Vehicle на карте, назначает (Статус: Assigned).
 * Водитель получает уведомление.
   * Вариант А: Отказывается (выбирает Reject_Reason). Заказ возвращается Диспетчеру (Статус: Created).
   * Вариант Б: Принимает (Статус: Accepted).
 * Водитель приезжает (Статус: Arrived).
 * Пассажир садится, водитель начинает движение (Статус: In_Progress).
 * Водитель привозит пассажира и нажимает кнопку (Статус: Completed).
 * Приложение генерирует QR-код со ссылкой вида: https://moment-lb.app/review/ORDER_ID.
 * Пассажир сканирует QR, оставляет Review.

8. Логика работы WebSockets (Real-time взаимодействие)
Для Next.js лучшим решением для WebSockets будет использование /websocket - service написаный на Vapor - я его уже использую в других проэктах и тестировал и 26000 сообщений в секунду держит без проблем.

Диспетчер должен видеть машины и заказы в реальном времени, а водитель должен мгновенно получать заявки.

8.1. Авторизация и Комнаты (Rooms)
При подключении по WebSocket клиент (браузер или PWA) передает JWT-токен. Сервер проверяет токен и подключает пользователя в соответствующие "комнаты":
 * room:dispatchers — сюда подключаются все диспетчеры. В эту комнату сервер шлет обновления координат ВСЕХ машин и новые заказы.
 * room:driver_{driver_id} — персональная комната водителя (для получения личных заказов).
 * room:order_{order_id} — комната конкретного заказа (для отслеживания его статуса).

11.2. События от Водителя (Client -> Server)
 * driver:location_update
   * Триггер: Срабатывает каждые N секунд (или метров), пока водитель на линии.
   * Payload: { driver_id, vehicle_id, lat, lng, heading }.
   * Действие сервера: Сервер сохраняет позицию в кэш (Redis) для быстрого доступа и делает broadcast (рассылку) события dispatcher:map_update в комнату room:dispatchers.
 * driver:status_change
   * Триггер: Водитель нажал "Выйти на линию" / "Уйти с линии".
   * Действие сервера: Обновляет статус водителя в БД, уведомляет диспетчеров.
 * order:accept / order:reject
   * Триггер: Водитель нажал кнопку в модальном окне нового заказа.
   * Действие сервера: Меняет статус в БД. Если отказ — записывает причину в Order_Reject, возвращает статус заказа в CREATED и отправляет алерт диспетчеру: dispatcher:order_rejected.
 * order:status_update
   * Триггер: Водитель меняет этапы поездки ("На месте", "Начал", "Завершил").
11.3. События от Диспетчера / Системы (Server -> Client)
 * driver:new_order
   * Триггер: Диспетчер назначил заказ водителю.
   * Цель: Отправляется в room:driver_{driver_id}.
   * Действие клиента (PWA): Показывает полноэкранный Pop-up с таймером, проигрывает звуковой сигнал.
 * dispatcher:map_update
   * Триггер: Сервер агрегирует координаты от всех активных водителей (например, раз в 2-3 секунды) и шлет пачкой диспетчерам.
   * Действие клиента (Web): Плавное перемещение маркеров машинок на карте Google/Yandex.
 * dispatcher:new_client_order
   * Триггер: Клиника создала новый заказ через веб-интерфейс (по HTTP REST API).
   * Действие сервера: После записи в БД, сервер кидает это событие по WebSocket всем диспетчерам.
   * Действие клиента (Web): В левой панели диспетчера появляется новая карточка заказа, звучит сигнал.

11.4. Сценарий обрыва связи (Офлайн/Реконнект)
Так как это мобильный интернет (3G/LTE/Е), обрывы WebSocket-соединения будут происходить регулярно.
 * На клиенте (PWA Водителя): * Если Socket отключается, PWA начинает складывать GPS-координаты и события (например, смену статуса заказа) во внутреннюю очередь (Local Array / IndexedDB).
   * Библиотека Socket.io автоматически пытается переподключиться.
 * При реконнекте:
   * PWA первым делом отправляет событие driver:sync_offline_data и передает весь массив накопленных координат и действий.
   * Сервер обрабатывает их хронологически (по timestamp), закрывая "дыры" в треке на карте диспетчера и обновляя актуальный статус заказа, если водитель завершил его будучи в офлайне (например, в подземном паркинге больницы).


Service Worker (SW) /home/hronop/node/moment/public/sw.js — это, по сути, прокси-сервер, который работает в фоновом режиме между вашим веб-приложением (PWA), браузером и сетью. В контексте корпоративного сервиса такси он является «сердцем» приложения водителя, отвечая за его живучесть в условиях плохой связи.
Вот 4 главные задачи, которые должен выполнять Service Worker в вашем проекте:
1. Кэширование статики и обеспечение работы без сети (Offline-first)
Водитель может оказаться в зоне без интернета (подземный паркинг клиники, загородная трасса). Приложение не должно показывать «динозавра» (ошибку отсутствия сети).
 * Установка (Install): При первом заходе водителя в PWA, Service Worker кэширует так называемый App Shell (HTML-каркас, CSS-стили, базовые JS-скрипты Next.js, иконки, шрифты).
 * Перехват запросов (Fetch): SW перехватывает все сетевые запросы приложения. Если интернета нет, он отдает закэшированный интерфейс. Водитель по-прежнему видит свои кнопки, просто сверху появляется плашка «Нет сети».
2. Фоновая синхронизация данных (Background Sync API)
Это критически важная функция для сохранения целостности данных о поездке.
 * Очередь запросов: Если водитель нажал кнопку «Завершить поездку» или система собрала новые GPS-координаты, но интернета в этот момент нет, запрос на сервер завершится ошибкой.
 * Задача SW: Service Worker перехватывает эту неудачную попытку, сохраняет данные (payload) локально в базе данных браузера (IndexedDB) и регистрирует задачу на синхронизацию.
 * Отправка: Как только смартфон снова ловит сеть, Service Worker в фоне (даже если водитель свернул приложение) достает данные из IndexedDB и отправляет их на сервер, закрывая заказ.
3. Обработка Push-уведомлений (Push API)
Диспетчер назначил заказ, и водитель должен узнать об этом моментально, даже если он смотрит YouTube или телефон лежит с выключенным экраном (в кармане).
 * Прием сигнала: SW слушает событие push от сервера (через сервисы вроде Firebase Cloud Messaging или VAPID ключи).
 * Отображение: Получив сигнал, Service Worker вызывает системное уведомление телефона (Notification API) с текстом: «Новый заказ! Клиника №5». При клике на это уведомление Service Worker разворачивает (или открывает заново) PWA на экране принятия заказа.
4. Управление обновлениями версий (Update Management)
Вы выкатили новую версию интерфейса (например, добавили новую кнопку в меню). PWA по умолчанию сильно кэшируется и может не показать изменения.
 * Задача SW: Периодически проверять сервер на наличие нового байт-кода.
 * Действие: Если есть новая версия, SW скачивает ее в фоне, а пользователю показывает небольшое уведомление: «Доступна новая версия приложения. Обновить?». При согласии SW подменяет старый кэш на новый.
Важное ограничение (Candor check):
Вы должны знать, что Apple исторически сильно ограничивает PWA. Если на Android (Chrome) Service Worker работает идеально, то на iOS (Safari):
 * Push-уведомления работают только начиная с iOS 16.4 и только если водитель добавил иконку PWA на главный экран (Add to Home Screen).
 * Фоновая синхронизация (Background Sync API) на iOS до сих пор поддерживается плохо, и при закрытии браузера процессы убиваются гораздо агрессивнее, чем на Android.
