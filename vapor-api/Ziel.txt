Задание: Миграция API-слоя с Next.js на Swift Vapor в высоконагруженном Docker-стеке.

Контекст:
У меня есть проект на Next.js (App Router), который работает в Docker на сервере Ubuntu 24.04 (8 ядер AMD Milan, 16GB RAM). В качестве базы данных используется PostgreSQL, а для хранения файлов — SeaweedFS. Текущий API на Next.js под нагрузкой выдает около 55 RPS с задержкой 900мс, и я хочу вынести критически важную логику на Swift Vapor, так как у меня уже есть работающий WebSocket-сервер на этом фреймворке.

Технические детали:


Nginx: Фронтенд остается на Next.js (порт 3002), но запросы к /api/* должны проксироваться на Swift Vapor (порт 8080).


SeaweedFS: Нужно реализовать логику работы с приватными файлами через заголовок X-Accel-Redirect, как это сейчас делает Nginx для локального хранилища.


Безопасность: В Swift-приложении нужно учитывать заголовки X-Real-IP и X-Forwarded-For, передаваемые через Nginx.

Что нужно сделать (задачи):

- Напиши архитектуру контроллера на Swift Vapor, который подключается к существующей базе Postgres.

- Реализуй метод API, который проверяет авторизацию (через JWT или сессию) и отдает команду Nginx на скачивание файла из SeaweedFS через X-Accel-Redirect.

- Настрой Dockerfile для многоэтапной сборки (multi-stage build), оптимизированный под минимальный размер финального образа (на базе Alpine или Amazon Linux 2).

- Обнови мой текущий конфиг Nginx, чтобы корректно разделить трафик между Next.js (SSR) и Swift API.
- должна быть новая переменная в .env которая будет определять с каким api (текущей Next.js API Routes или Swift ) работать приложению 
Требования к коду:

Использование Swift 6.0.

Асинхронность через async/await.

Типизация моделей через Fluent (для Postgres) и Content.
